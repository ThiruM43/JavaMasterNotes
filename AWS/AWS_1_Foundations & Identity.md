ஆமா, நைட்டு பூரா உக்காந்து லாக் [Log - நம்ம ஆப்ல என்னென்ன நடக்குதுன்னு ரெக்கார்ட் ஆகுற இடம்] செக் பண்ணி கடைசியா அந்த ப்ராப்பர்ட்டீஸ் ஃபைல கண்டுபிடிச்சு சர்வர ரீஸ்டார்ட் [Server restart - சர்வரை ஆஃப் பண்ணி ஆன் பண்றது] பண்ணிருப்பீங்க. ஒரு 12 வருஷம் ஜாவா பேக்கெண்ட் டெவலப்பரா [Java Backend Developer - ஜாவா வச்சு சர்வர் சைடுக்கு கோட் எழுதுறவங்க] இருக்குறவங்களுக்கு கண்டிப்பா இந்த மாதிரி ஒரு நைட்மேர் எக்ஸ்பீரியன்ஸ் இருக்கும்.

எக்ஸாக்ட்லி. ஆனா பாத்தீங்கன்னா, இதே இன்ப்ராஸ்ட்ரக்சர நாம க்ளவுடுக்கு [Cloud - இன்டர்நெட்ல இருக்கிற சர்வர்ஸ்] கொண்டு போகும்போது, இந்த மாதிரி க்ரெடென்ஷியல்ஸ ஹார்ட்கோட் பண்றது [Hardcode credentials - பாஸ்வேர்ட், சீக்ரெட் கீஸ் எல்லாத்தையும் டைரக்டா கோடுலயே எழுதி வைக்கிறது], அப்புறம் பிசிக்கல் சர்வர் ரேம் அப்கிரேட் பண்ண [Physical server RAM upgrade - நம்ம கண்ணுக்கு தெரியுற சர்வர் மெஷின்ல ரேம் அதிகப்படுத்துறது] டவுன்டைம் [Downtime - அப்ளிகேஷன் ஒர்க் ஆகாம ஆஃப் ஆகி இருக்குற நேரம்] எடுக்குறது, இதெல்லாம் ஆக்சுவலா முற்றிலுமா மாறிடுது.

கரெக்ட். இன்னைக்கு நாம ஸ்டீபன் மாரேகோவோட AWS Cloud [AWS Cloud - அமேசானோட க்ளவுட் சர்வீஸ்], EC2 [EC2 - க்ளவுட்ல நாம வாங்குற வாடகை கம்ப்யூட்டர்] அப்புறம் IAM ரிசோர்சஸ [IAM resources - யாருக்கு எந்த ஆக்சஸ் கொடுக்கணும்னு முடிவு பண்ற சர்வீஸ்] வச்சு ஒரு விரிவான அலசல் பண்ண போறோம்.

நிச்சயமா. நீங்க தினமும் யூஸ் பண்ற உங்களுக்கு ரொம்பவே அத்துப்படியான ஸ்ப்ரிங் பூட் [Spring Boot - ஜாவால ஈஸியா ஆப் உருவாக்க யூஸ் பண்ற ஒரு பிரேம்வொர்க்], டாம்கேட் [Tomcat - ஜாவா வெப் ஆப்ஸ் ஓடுற ஒரு சர்வர்], ஜேவிஎம் [JVM - ஜாவா கோட ரன் பண்ற ஒரு மெஷின்] மாதிரியான ஜாவா கான்செப்ட்ஸ வச்சே AWS ஓட கோர் இன்ப்ராஸ்ட்ரக்சர உங்களோட மொழியிலேயே இன்னைக்கு டீகோட் பண்ண போறோம்.

ஒரு ட்ரெடிஷனல் ஆன்-ப்ரமைஸ் இன்ப்ராஸ்ட்ரக்சர்ல [On-premise infrastructure - நம்ம கம்பெனிக்குள்ளயே பிசிக்கலா சர்வர்ஸ் வச்சு மெயின்டைன் பண்றது] நீங்க வந்து ஜேவிஎம் டியூனிங் [JVM tuning - ஜேவிஎம் வேகமா ஓட செட் பண்றது], கார்பேஜ் கலெக்ஷன் [Garbage collection - தேவையில்லாத மெமரியை ஆட்டோமேட்டிக்கா கிளீன் பண்றது], அப்புறம் ஜேபாஸ் த்ரெட் பூல்ஸ்னு [JBoss thread pools - ஒரே நேரத்துல பல வேலைகளை செய்ய த்ரெட்ஸ் ஒதுக்குறது] பேர் மெட்டல் ஹார்ட்வேர் [Bare metal hardware - எந்த விர்ச்சுவல் லேயரும் இல்லாத நேரடியான பிசிக்கல் சர்வர்] கூட மல்லு கட்டிருப்பீங்க. ஆனா AWS பொறுத்தவரைக்கும் இந்த ஹார்ட்வேர் லெவல் அப்ஸ்ட்ராக்ஷன [Hardware level abstraction - ஹார்ட்வேர் எப்படி வேலை செய்யுதுன்னு நாம கவலைப்படாம யூஸ் பண்றது] அவங்க முழுசா கையாள்றாங்க.

ஆமா, அது ரொம்ப பெரிய ரிலீஃப். நீங்க சோர்சஸ்ல ஷேர் பண்ணிருக்கற IAM, EC2, EPS [EPS - இங்க EBS அதாவது எலாஸ்டிக் பிளாக் ஸ்டோரேஜ் பத்தி தான் தப்பா சொல்லிருக்காங்கனு நினைக்கிறேன்] இந்த மூணுமே ஒரு என்டர்பிரைஸ் ஜாவா அப்ளிகேஷன [Enterprise Java Application - பெரிய கம்பெனிகள் யூஸ் பண்ற பெரிய ஜாவா சாப்ட்வேர்] க்ளவுட்ல டிப்ளாய் பண்றதுக்கான [Deploy - ஆப்ப சர்வர்ல போட்டு ரன் பண்றது] செக்யூரிட்டி [Security - பாதுகாப்பு], கம்ப்யூட் [Compute - ப்ராசஸ் பண்ற பவர்] மற்றும் ஸ்டோரேஜ் [Storage - டேட்டாவ சேவ் பண்ற இடம்] தூண்கள்.

கண்டிப்பா. இத நாம வெறும் தியரியா மட்டும் பார்க்காம நிஜத்துல ஒரு ஜாவா ஆர்கிடெக்சர்ல [Java Architecture - ஜாவா ஆப் எப்படி டிசைன் பண்ணிருக்காங்கனு காட்ற ஸ்ட்ரக்சர்] இது எப்படி பொருந்தும்னு பிரிச்சு மேய்வோம்.

சூப்பர். அப்போ மொதல்ல செக்யூரிட்டில இருந்தே ஆரம்பிக்கலாம். இப்போ நாம ஒரு மோனோலித்திக் ஸ்ப்ரிங் பூட் அப்ளிகேஷன் [Monolithic Spring Boot application - எல்லா பீச்சர்ஸும் ஒரே கோட் பேஸ்ல இருக்குற ஒரு பெரிய ஆப்] எழுதுறோம்னா, எந்த எண்ட் பாயிண்ட்ட [End point - ஆப்ப கனெக்ட் பண்ற ஒரு URL] யார் ஆக்சஸ் பண்ணனும்னு ஸ்ப்ரிங் செக்யூரிட்டி [Spring Security - ஸ்ப்ரிங் ஆப்புக்கு செக்யூரிட்டி குடுக்குற ஒரு டூல்] யூஸ் பண்ணி செட் பண்ணுவோம் இல்லையா?

ஆமா, அந்த ப்ரீ-ஆத்தரைஸ் அனொடேஷன்லாம் [Pre-authorize annotation - யாருக்கு பர்மிஷன் இருக்குனு செக் பண்ற கோடு டேக்] யூஸ் பண்ணுவோம்.

கரெக்ட். ப்ரீ-ஆத்தரைஸ், ஹேஸ் ரோல் அட்மின் [hasRole('ADMIN') - அட்மின் ரோல் இருக்கானு செக் பண்றது] அப்படின்னு ரோல்-பேஸ்ட் ஆக்சஸ் கண்ட்ரோல் [Role-based access control - அவங்க ரோலுக்கு ஏத்த மாதிரி ஆக்சஸ் கொடுக்கிறது] அதாவது ஆர்பேக் (RBAC) செட் பண்ணுவோம். அப்போ ஒரு அக்கவுண்ட் லெவல்ல AWS-அ சுத்தி ஒரு பெரிமீட்டர் செக்யூரிட்டி [Perimeter security - நம்ம சிஸ்டம்க்கு வெளியில போடுற பாதுகாப்பு வளையம்] போடணும்னா, ஸ்ப்ரிங் செக்யூரிட்டிக்கு நிகரான ஒரு விஷயம் அங்க என்னவா இருக்கும்?

அங்க தான் IAM அதாவது ஐடென்டிட்டி அண்ட் ஆக்சஸ் மேனேஜ்மென்ட் [Identity and Access Management - யார் யாருக்கு என்னென்ன பர்மிஷன் தரணும்னு கண்ட்ரோல் பண்ற சர்வீஸ்] உள்ள வருது. நீங்க ஸ்ப்ரிங் செக்யூரிட்டில பண்ற அதே ஆர்பேக் கான்செப்ட் தான், ஆனா மொத்த AWS இன்ப்ராஸ்ட்ரக்சருக்குமான கீ கீப்பர் இதுதான்.

ஓஹோ, அப்ப இது ஒரு குளோபல் சர்வீஸ் [Global service - எந்த குறிப்பிட்ட ரீஜனுக்கும் கட்டுப்படாம மொத்த AWS-க்கும் பொதுவா வேலை செய்யுற சர்வீஸ்] மாதிரி செயல்படுது.

IAM-ல யூசர்ஸ் [Users - ஆட்களோ இல்ல வேற சிஸ்டம்ஸோ] அப்படிங்கறது உங்க கம்பெனில இருக்கிற நிஜமான டெவலப்பர்ஸ் அல்லது சிஸ்டம்ஸ். அப்புறம் குரூப்ஸ் [Groups - பல யூசர்ஸ ஒன்னா சேர்த்து ஒரே பர்மிஷன் குடுக்குற அமைப்பு] அப்படிங்கறது அந்த யூசர்ஸ ஒன்னா சேர்க்கிற ஒரு லாஜிக்கல் அமைப்பு. உதாரணத்துக்கு பேக்கெண்ட் டெவ் குரூப் [Backend dev group - பேக்கெண்ட் டெவலப்பர்ஸ்க்குனு இருக்குற குரூப்] அல்லது டிபிஏ குரூப் [DBA group - டேட்டாபேஸ் அட்மின்ஸ்க்குனு இருக்குற குரூப்].

ஓகே. யூசர்ஸ் அண்ட் குரூப்ஸ் புரிஞ்சது. ஆனா இவங்களுக்கு பர்மிஷன்ஸ் எப்படி அசைன் பண்றது? ஸ்ப்ரிங்ல நாம ஜாவா கோடு வழியா பண்ணுவோம்.

இங்க வந்து AWS பாலிசிஸ் [AWS Policies - யாருக்கு என்னென்ன உரிமை இருக்குனு எழுதி வச்சிருக்கற டாக்குமெண்ட்] அப்படிங்கிற ஒரு விஷயத்தை யூஸ் பண்றாங்க. இந்த பாலிசிஸ் எல்லாமே ஆக்சுவலா ஜேசன் ஃபார்மட்ல [JSON format - கம்ப்யூட்டருக்கு புரியுற மாதிரி கீ-வேல்யூ பேரா டேட்டாவ எழுதுற முறை] தான் இருக்கும். நீங்க ஜேசன்ல தான் ரூல்ஸ் எழுதணும்.

ஆமா, அந்த ஜேசன் பாலிசிஸ நான் பார்த்திருக்கேன். அதுல எஃபெக்ட் [Effect - அனுமதி குடுக்கலாமா வேணாமானு சொல்றது (Allow/Deny)], ஆக்சன் [Action - என்ன வேலை செய்ய போறாங்கனு சொல்றது], ரிசோர்ஸ் [Resource - எந்த சர்வர் இல்ல டேட்டாபேஸ்ல வேலை செய்ய போறாங்கனு சொல்றது] அப்படின்னு மூணு கீஸ் இருக்கும். ஆனா எனக்கு ஒரு சந்தேகம். இவ்வளவு பெரிய க்ளவுட் என்விரான்மென்ட்ல கோடிக்கணக்கான ஏபிஐ கால்ஸ் [API calls - ஒரு சாப்ட்வேர் இன்னொரு சாப்ட்வேர் கிட்ட ரெக்வெஸ்ட் குடுக்குறது] நடக்கும்போது, இந்த ஜேசனை வச்சே AWS எப்படி ஆக்சஸை இவாலுவேட் [Evaluate - சரிபார்க்கிறது] பண்ணுது?

அது ஒரு நல்ல கேள்வி. இந்த இடத்துல ஏதாவது லூப்ஹோல் [Loophole - தப்பு பண்ண வழி இருக்குற இடம்] வர வாய்ப்பு இருக்கான்னு நீங்க யோசிக்கலாம். ஆனா அந்த இவாலுவேஷன் ப்ராசஸ்ல தான் AWS ஓட ஒரு மிக முக்கியமான செக்யூரிட்டி மெக்கானிசம் இருக்கு. அதுக்கு பேரு இம்ப்ளிசிட் டினை [Implicit deny - வெளிப்படையா பர்மிஷன் கொடுக்கலனா, ஆட்டோமேட்டிக்கா பிளாக் பண்றது].

இம்ப்ளிசிட் டினை அப்படின்னா டீபால்ட்டா [Default-a - ஆரம்பத்துல இருந்தே] எல்லாமே பிளாக் ஆயிருக்குமா?

ஆமா. ஒரு IAM பாலிசில நீங்க வெளிப்படையா இந்த யூசருக்கு ஈசி2வை ஆக்சஸ் பண்ண அலவ் [Allow - அனுமதி] கொடு அப்படின்னு எஃபெக்ட் செட் பண்ணலன்னா, AWS தானாகவே அந்த ரெக்வெஸ்ட்ட டினை [Deny - மறுக்கிறது] பண்ணிடும். நீங்க எதுவும் சொல்லலன்னா டீபால்ட் பதில் நோ தான்.

வாவ், செம ஸ்ட்ரிக்ட். அப்போ ஆக்சன் அப்படிங்கிறது எதை குறிக்கும்?

ஆக்சன் அப்படிங்கிறது அவங்க என்ன ஏபிஐ கால் பண்ணலாம்னு குறிக்கும். உதாரணத்துக்கு ஈசி2 கோலன் ஸ்டார்ட் இன்ஸ்டன்சஸ் [EC2 Start Instances - ஈசி2 சர்வரை ஸ்டார்ட் பண்ற கமாண்ட்]. அப்புறம் ரிசோர்ஸ் அப்படிங்கிறது எந்த குறிப்பிட்ட சர்வர் மேல இதை பண்ணலாம்னு சொல்லும்.

புரிஞ்சது. இந்த ஜேசன் பாலிசிஸ் மூலமா லீஸ்ட் பிரிவிலிஜ் பிரின்சிபிளை [Least Privilege Principle - ஒரு வேலை செய்ய எவ்வளவு குறைஞ்ச ஆக்சஸ் தேவையோ அதை மட்டும் கொடுக்கிறது] AWS மிக கடுமையா என்போர்ஸ் [Enforce - கண்டிப்பா ஃபாலோ பண்ண வைக்கிறது] பண்ணுது. இப்போ என்னோட அந்த 2014 நைட்மேருக்கு வருவோம்.

சொல்லுங்க.

என் ஸ்ப்ரிங் பூட் ஆப்ல டேட்டாபேஸ் யூசர்நேம், பாஸ்வேர்டை ப்ராப்பர்ட்டி ஃபைல்ல வைப்பேன். இப்போ AWS-ல ஒரு ஈசி2 இன்ஸ்டன்ஸ்ல ஓடுற டாம்கேட் சர்வர், அங்க இருக்குற ஒரு எஸ்3 பக்கெட்டையோ [S3 bucket - பைல்ஸ ஸ்டோர் பண்ற க்ளவுட் ஸ்டோரேஜ்] அல்லது ஆர்டிஎஸ் டேட்டாபேஸையோ [RDS Database - ரிலேஷனல் டேட்டாபேஸ் சர்வீஸ், அதாவது டேபிள் டேபிளா டேட்டா சேமிக்கிற இடம்] ஆக்சஸ் பண்ணனும்னா, அந்த க்ரெடென்ஷியல்ஸ [Credentials - லாகின் ஐடி, பாஸ்வேர்ட்] எங்க வைக்கிறது?

மறுபடியும் ப்ராப்பர்ட்டி ஃபைல்ல ஹார்ட்கோட் பண்ணா அது பெரிய செக்யூரிட்டி ரிஸ்க் ஆச்சேன்னு யோசிக்கிறீங்க, கரெக்டா?

ஆமா, கண்டிப்பா ரிஸ்க் தான்.

அந்த ரிஸ்க்கை தவிர்க்கிறதுக்கு தான் AWS IAM ரோல்ஸ் [IAM Roles - யூசருக்கு பதிலா ரிசோர்ஸ்க்கு கொடுக்கிற பர்மிஷன் ஐடி] அப்படிங்கிற ஒரு கான்செப்ட்டை வடிவமைச்சிருக்காங்க. ஒரு யூசருக்கு பர்மிஷன் கொடுக்கிற மாதிரி, உங்க ஈசி2 இன்ஸ்டன்ஸுக்கே, அதாவது அந்த விர்ச்சுவல் மெஷினுக்கே [Virtual Machine - பிசிக்கலா இல்லாம சாப்ட்வேர் மூலமா ஓடுற கம்ப்யூட்டர்], நீங்க ஒரு ரோலை அசைன் பண்ணிடலாம்.

ஒரு நிமிஷம், மெஷினுக்கே ரோல் கொடுக்கலாமா? அது எப்படி ஒர்க் ஆகும்?

இது எப்படின்னா, ஈசி2 இன்ஸ்டன்ஸ்க்குள்ள AWS ஓட இன்ஸ்டன்ஸ் மெட்டாடேட்டா சர்வீஸ் [Instance Metadata Service - நம்ம ஈசி2 சர்வரை பத்தின டீடைல்ஸ் கொடுக்கிற ஒரு சர்வீஸ்] அப்படின்னு ஒண்ணு ஓடிட்டு இருக்கும். அது பேக்ரவுண்ட்ல AWS கிட்ட பேசி, டெம்பரரியான க்ரெடென்ஷியல்ஸ வாங்கிக்கிடும்.

அப்போ அந்த க்ரெடென்ஷியல்ஸ் பெர்மனன்ட் [Permanent - நிரந்தரமான] இல்லையா?

இல்ல, இல்ல. இந்த க்ரெடென்ஷியல்ஸ் கொஞ்ச நேரத்துக்கு ஒரு தடவை தானாகவே ரொட்டேட் [Rotate - மாறிக்கிட்டே இருக்குறது] ஆயிடும். அதனால நீங்க உங்க ஜாவா கோடுல எந்த ஒரு சீக்ரெட் கீயையோ, பாஸ்வேர்டையோ ஹார்ட்கோட் பண்ணவே தேவையில்லை.

இது செம ஃபீச்சர். அப்போ ஸ்ப்ரிங் செக்யூரிட்டில ஒரு குறிப்பிட்ட ஏபிஐ எண்ட் பாயிண்ட்ட செக்யூர் பண்ற மாதிரி, IAM ரோல்ஸ் மூலமா ஒரு முழு சர்வரையுமே செக்யூர் பண்ணிடலாம்.

எக்ஸாக்ட்லி.

இப்போ ஒரு சொல்யூஷன்ஸ் ஆர்கிடெக்ட் அசோசியேட் அதாவது SAA எக்ஸாம் [SAA exam - AWS-ல நாம ஆர்கிடெக்ட்னு சர்டிபிகேட் வாங்குறதுக்கான ஒரு பரீட்சை] சினாரியோவா யோசிப்போம். ஒரு கம்பெனில 100 டெவலப்பர்ஸ் இருக்காங்க. யார் யாரை எந்தெந்த ரிசோர்சஸை ஆக்சஸ் பண்றாங்க, யாரோட ஆக்சஸ் கீஸ் எல்லாம் ரொம்ப நாளா யூஸ் பண்ணாம ஐடிலா [Idle-a - சும்மாவே இருக்குறது] இருக்குன்னு எப்படி ட்ராக் பண்றது? ஏன்னா என்டர்பிரைஸ்ல ஆளுங்க வருவாங்க, போவாங்க இல்லையா?

கண்டிப்பா, SAA எக்ஸாம்ல இது அடிக்கடி வர்ற ஒரு கேள்வி. இதுக்கு ரெண்டு ஆடிட் டூல்ஸ் [Audit tools - என்னென்ன நடந்திருக்குனு செக் பண்ற டூல்ஸ்] AWS-ல இருக்கு. ஒன்னு பார்த்தீங்கன்னா, IAM க்ரெடென்ஷியல்ஸ் ரிப்போர்ட் [IAM Credentials Report - எல்லா யூசரோட பாஸ்வேர்ட், ஆக்சஸ் கீ ஸ்டேட்டஸை காட்ற ரிப்போர்ட்].

ஓகே, இது எப்படி வேலை செய்யும்?

இது அக்கவுண்ட் லெவல்ல செயல்படும். உங்க அக்கவுண்ட்ல இருக்கிற எல்லா யூசர்ஸோட பாஸ்வேர்ட்ஸ் மற்றும் ஆக்சஸ் கீஸ எப்போ கடைசியா ரொட்டேட் ஆச்சு அப்படிங்கிற மொத்த லிஸ்ட்டையும் ஒரு சிஎஸ்வி ஃபைலா [CSV file - கமா-வால் பிரிக்கப்பட்ட டேட்டா இருக்குற பைல், எக்செல் மாதிரி] கொடுத்துரும்.

சூப்பர், இன்னொரு டூல் என்ன சொன்னீங்க?

இன்னொன்னு வந்து IAM ஆக்சஸ் அட்வைசர் [IAM Access Advisor - ஒரு யூசர் எந்த சர்வீஸை கடைசியா எப்ப யூஸ் பண்ணாருனு சொல்ற டூல்]. இது யூசர் லெவல்ல செயல்படும். ஒரு குறிப்பிட்ட யூசர் எந்தெந்த AWS சர்வீசஸை கடைசியா எப்போ ஆக்சஸ் பண்ணாங்கன்னு காட்டும்.

அப்போ ஒரு டெவலப்பர் கடந்த 90 நாளா எஸ்3யை யூஸ் பண்ணவே இல்லைன்னா, இந்த ஆக்சஸ் அட்வைசரை வச்சு கண்டுபிடிச்சிரலாமா?

ஆமா, அதை கண்டுபிடிச்சு அந்த பர்மிஷனை நீங்க பாலிசில இருந்து தூக்கிடலாம். செக்யூரிட்டி பெர்ஸ்பெக்டிவ்ல [Security perspective - பாதுகாப்பு கோணத்துல] இது ரொம்ப முக்கியம்.

IAM அக்கவுண்ட் லெவல் செக்யூரிட்டியை கவனிச்சுக்குதுன்னு தெளிவா புரியுது. அடுத்தபடியா, நம்ம ஜேவிஎம்மை எங்க ரன் பண்றதுன்னு பார்ப்போம். வழக்கமா பேர் மெட்டல் சர்வர்ஸ்ல ரேமையும், சிபியு கோர்ஸையும் கணக்கு போட்டு அப்ளிகேஷனை டெப்லாய் பண்ணுவோம்.

கரெக்ட். AWS-ல EC2, அதாவது எலாஸ்டிக் கம்ப்யூட் கிளவுட் [Elastic Compute Cloud - நம்ம தேவைக்கு ஏத்த மாதிரி கூட்டிக்கவோ குறைச்சுக்கவோ கூடிய க்ளவுட் கம்ப்யூட்டர்] வழியா விர்ச்சுவல் மெஷின்ஸ் வாங்குறோம். ஆனா இங்க ஹார்ட்வேர் அப்ஸ்ட்ராக்ஷன் இருக்கறதால, ஒரு ஜாவா டெவலப்பரா என் ஒர்க்லோடுக்கு எந்த ஈசி2 இன்ஸ்டன்ஸ் டைப் சரியா இருக்கும்னு எப்படி முடிவு பண்றது?

நீங்க பிசிக்கல் சர்வர் வாங்கும்போதே, இந்த ப்ராஜெக்ட்டுக்கு அதிக ரேம் தேவைப்படுமா, இல்ல சிபியு தேவைப்படுமா அப்படின்னு யோசிப்பீங்க இல்லையா? அதே லாஜிக் தான் இங்கயும். AWS இதை இன்ஸ்டன்ஸ் ஃபேமிலிஸ் [Instance families - வேலைக்கு ஏத்த மாதிரி சர்வர் வகைகளை பிரிச்சு வச்சிருக்கிறது] அப்படின்னு பிரிக்கிறாங்க. இதை ஈசியா ஞாபகம் வச்சுக்க சில நிமோனிக்ஸ் [Mnemonics - ஈஸியா ஞாபகம் வச்சுக்க யூஸ் பண்ற ஷார்ட்கட்ஸ்] இருக்கு.

நிமோனிக்ஸா? சொல்லுங்க கேட்போம்.

C for Compute. அதாவது கம்ப்யூட் ஆப்டிமைஸ்ட். இது C ஃபேமிலி. உங்க ஜாவா ஆப்ல ஸ்ப்ரிங் பேட்ச் யூஸ் பண்ணி லட்சக்கணக்கான ரெக்கார்ட்ஸை ப்ராசஸ் பண்றீங்க, அல்லது மீடியா டிரான்ஸ்கோடிங் [Media transcoding - ஒரு வீடியோ/ஆடியோ பார்மெட்டை இன்னொரு பார்மெட்டுக்கு மாத்துறது] பண்றீங்கன்னா, சிபியு கோர்ஸ் அதிகமா தேவைப்படும். அங்க C இன்ஸ்டன்சஸை சூஸ் பண்ணனும்.

ஓகே, C for Compute. அப்புறம்?

அடுத்தது R for RAM. மெமரி ஆப்டிமைஸ்ட். அதாவது R ஃபேமிலி.

ஒரு நிமிஷம், மெமரி ஆப்டிமைஸ்ட் அப்படின்னா... இப்போ நான் ஹேசல் காஸ்ட் அல்லது ரெடிஸ் மாதிரி ஒரு டிஸ்ட்ரிபியூட்டட் இன்-மெமரி கேஷை [In-memory cache - டேட்டாவை ஹார்ட் டிஸ்க்ல சேவ் பண்ணாம வேகமா எடுக்க ரேம்லயே சேவ் பண்றது] என் ஆர்கிடெக்சர்ல யூஸ் பண்றேன்னு வைங்க. அங்க டேட்டா எல்லாமே ரேம்ல தான் உக்காந்து இருக்கும். அப்போ அந்த மாதிரி கேஷிங் சர்வர்ஸ்க்கு R ஃபேமிலி தான் சரியான சாய்ஸா இருக்குமா?

எக்ஸாக்ட்லி. இன்-மெமரி டேட்டாபேசஸ், பெரிய பிக் டேட்டா அனலிட்டிக்ஸ் இன்ஜின், இது எல்லாத்துக்கும் ரேம் தான் பாட்டில்நெக் [Bottleneck - வேகம் குறைய காரணமா இருக்குற இடம்]. அதனால R ஃபேமிலி தான் பெஸ்ட். அடுத்தது I for IO [IO - Input/Output, அதாவது ரீட்/ரைட் பண்றது]. அதாவது ஸ்டோரேஜ் ஆப்டிமைஸ்ட், I ஃபேமிலி.

ஸ்டோரேஜ் ஆப்டிமைஸ்டுனா... ஹார்ட் டிஸ்க் ஸ்பீட் அதிகமா இருக்குமா?

ஆமா, இதுல லோக்கல் டிஸ்க் பெர்ஃபார்மன்ஸ் மிக மிக வேகமா இருக்கும். அதனால நோ-எஸ்கியூஎல் டேட்டாபேசஸ் [NoSQL databases - டேபிள் பார்மெட்ல இல்லாம ஸ்ட்ரக்சர் இல்லாத டேட்டாவை சேவ் பண்றது], கசாண்ட்ரா மாதிரி அதிக ரீட் ரைட் நடக்குற இடங்கள்ல I இன்ஸ்டன்சஸை யூஸ் பண்ணுவாங்க.

புரிஞ்சது. அப்போ நம்ம சாதாரண வெப் சர்வருக்கு என்ன யூஸ் பண்றது?

அதுக்கு கடைசியா M அல்லது T ஃபேமிலி இருக்கு. இது ஜெனரல் பர்பஸ் [General purpose - எல்லா வகையான சாதாரண வேலைகளுக்கும் யூஸ் பண்றது]. உங்க சாதாரண டாம்கேட் வெப் சர்வருக்கு கம்ப்யூட், மெமரி எல்லாமே பேலன்ஸ்டா தேவைப்படும். அதுக்கு இந்த ஜெனரல் பர்பஸ் இன்ஸ்டன்சஸை யூஸ் பண்ணலாம். SAA எக்ஸாம்ல, ஒரு ஒர்க்லோடை கொடுத்துட்டு, இதுக்கு எந்த இன்ஸ்டன்ஸ் டைப் சரின்னு சினாரியோ பேஸ்டு கேள்விகள் கண்டிப்பா வரும்.

ஓகே. நம்ம ஜேவிஎம் ரன் ஆக ஈசி2 இன்ஸ்டன்ஸை செலக்ட் பண்ணியாச்சு. சிபியு, ரேம் எல்லாம் கிடைச்சிருச்சு. ஆனா நம்ம டாம்கேட் சர்வர் சும்மா ஓடாது. அது ஜிபி கணக்குல கார்பேஜ் கலெக்ஷன் லாக்ஸ் [Garbage collection logs - மெமரி கிளீன் ஆகுறப்போ உருவாகுற லாக் பைல்ஸ்] எழுதும். அப்ளிகேஷன் லாக்ஸ் இருக்கும்.

அது மட்டும் இல்லாம யூசர்ஸ் அப்லோட் பண்ற ஃபைல்ஸ் இருக்கும். ஈசி2 அப்படிங்கிறது வெறும் சிபியு, ரேம் மட்டும் தான்னா, இந்த டேட்டாவெல்லாம் எங்க ஸ்டோர் பண்றது? ஹார்ட் டிஸ்க்குக்கு இங்க என்ன ஆர்கிடெக்சர் இருக்கு?

இங்க தான் கிளவுடோட ஸ்டோரேஜ் அப்ஸ்ட்ராக்ஷன் [Cloud storage abstraction - பிசிக்கல் ஹார்ட் டிஸ்க் எப்படி வேலை செய்யுதுனு நம்ம கவலைப்படாம ஸ்டோரேஜ் யூஸ் பண்றது] வருது. ஈசி2க்கு குறிப்பா ரெண்டு வகையான பிளாக் ஸ்டோரேஜ் ஆப்ஷன்ஸ் இருக்கு. ஒன்னு இபிஎஸ் (EBS). அதாவது எலாஸ்டிக் பிளாக் ஸ்டோர் [Elastic Block Store - ஈசி2 சர்வருக்கு அட்டாச் பண்ற ஒரு நெட்வொர்க் ஹார்ட் டிஸ்க்]. இன்னொன்னு ஈசி2 இன்ஸ்டன்ஸ் ஸ்டோர் [EC2 Instance Store - சர்வருக்குள்ளயே டைரக்டா இருக்குற ஒரு டெம்பரரி ஸ்டோரேஜ்].

ஓகே, இபிஎஸ் அப்படின்னா என்ன?

முதல்ல இபிஎஸை எடுத்துப்போம். இது ஒரு நெட்வொர்க் டிரைவ் [Network drive - நெட்வொர்க் மூலமா கனெக்ட் ஆகுற ஸ்டோரேஜ்]. அதாவது, ஒரு பெரிய டேட்டா சென்டர்ல எங்கேயோ ஒரு ஸ்டோரேஜ் ஏரியா நெட்வொர்க்ல இருக்கிற ஒரு பிசிக்கல் டிஸ்க்கை, நெட்வொர்க் மூலமா உங்க ஈசி2 இன்ஸ்டன்ஸ்க்கு அட்டாச் பண்றாங்க. சிம்பிளா சொல்லணும்னா, ஒரு நெட்வொர்க் யுஎஸ்பி ஸ்டிக் மாதிரி இதை கற்பனை பண்ணிக்கோங்க.

நெட்வொர்க் மூலமா அட்டாச் பண்றாங்களா? அப்போ நான் லாக் ஃபைல்ல ஒரு வரியை எழுதினா கூட, அது நெட்வொர்க் வழியா டிராவல் பண்ணி தான் அந்த டிஸ்க்ல போய் ஸ்டோர் ஆகுமா? இதுல லேட்டன்சி [Latency - டேட்டா போய் சேர ஆகுற தாமதம்] இருக்காதா?

ஆக்சுவலா ஓரளவுக்கு லேட்டன்சி இருக்கும். ஆனா AWS அந்த நெட்வொர்க்கை மிக வேகமா ஆப்டிமைஸ் பண்ணிருக்காங்க. இதுல ஒரு மிகப்பெரிய லாபம் என்னன்னா, ஒருவேளை உங்க ஈசி2 இன்ஸ்டன்ஸ் கிராஷ் ஆகி டெர்மினேட் ஆனாலும், நெட்வொர்க்ல இருக்கிற அந்த இபிஎஸ் டிரைவ்ல உங்க டேட்டா பத்திரமா இருக்கும்.

ஓ, டேட்டா லாஸ் இருக்காது.

ஆமா, நீங்க வேற ஒரு புது ஈசி2 இன்ஸ்டன்ஸை உருவாக்கி, இந்த இபிஎஸை அதுல அட்டாச் பண்ணி வேலையை தொடரலாம். ஆனா இங்க தான் இன்னொரு ஸ்டோரேஜ் டைப் வருது. அதுதான் ஈசி2 இன்ஸ்டன்ஸ் ஸ்டோர். இது நெட்வொர்க் டிரைவ் கிடையாது.

அப்போ இது லோக்கல் டிரைவா?

ஆமா, உங்க ஈசி2 விர்ச்சுவல் மெஷின் எந்த பிசிக்கல் சர்வர் ரேக்ல ஓடுதோ, அந்த ரேக்லயே நேரடியாக இணைக்கப்பட்ட பிசிக்கல் ஹார்ட் டிரைவ் தான் இந்த இன்ஸ்டன்ஸ் ஸ்டோர்.

டைரக்ட் அட்டாச்டு பிசிக்கல் டிரைவா? அப்போ நெட்வொர்க் லேட்டன்சியே இருக்காது, ஐஓபிஎஸ் (IOPS) [IOPS - ஒரு செகண்டுக்கு எத்தனை தடவை டேட்டாவை ரீட்/ரைட் பண்ண முடியும்ங்கிற கணக்கு] அதாவது இன்புட் அவுட்புட் ஆப்பரேஷன்ஸ் ரொம்ப அதிகமா இருக்குமே?

கண்டிப்பா, ரொம்ப ஃபாஸ்ட்டா இருக்கும். ஆனா சோர்சஸ்ல இது எஃபிமரல் ஸ்டோரேஜ் [Ephemeral storage - கொஞ்ச நேரம் மட்டும் இருக்கக்கூடிய தற்காலிகமான ஸ்டோரேஜ்], அதாவது தற்காலிகமானது அப்படின்னு சொல்லிருக்காங்களே? பிசிக்கல் டிஸ்க்ல எழுதுற டேட்டா எப்படி அழியும்?

அங்க தான் AWS ஓட ஹைப்பர்வைசர் மெக்கானிசமை [Hypervisor mechanism - ஒரு பிசிக்கல் மெஷின்ல பல விர்ச்சுவல் மெஷின்ஸ் ஓட வைக்கிற சாப்ட்வேர்] நீங்க புரிஞ்சுக்கணும். உங்க ஈசி2 இன்ஸ்டன்ஸை நீங்க ஸ்டாப் பண்ணிட்டு மறுபடியும் ஸ்டார்ட் பண்ணும்போது, AWS உங்களோட விர்ச்சுவல் மெஷினை அதே பிசிக்கல் ரேக்ல தான் ஸ்டார்ட் பண்ணனும்னு எந்த அவசியமும் இல்லை.

ஓ, வேற சர்வருக்கு மாத்திடுவாங்களா?

ஆமா, ரிசோர்ஸ் ஆப்டிமைசேஷனுக்காக டேட்டா சென்டர்ல இருக்கிற வேறொரு பிசிக்கல் ரேக்குக்கு உங்க விஎம்மை அவங்க மாற்றலாம். உங்க விஎம் புது ரேக்குக்கு மாறிடும், ஆனா பழைய ரேக்ல இருந்த அந்த இன்ஸ்டன்ஸ் ஸ்டோர் பிசிக்கல் ஹார்ட் டிரைவ் அங்கேயே தங்கிடும். அதனால தான் ஈசி2வை ஸ்டாப் ஸ்டார்ட் பண்ணா, இன்ஸ்டன்ஸ் ஸ்டோர்ல இருக்கிற டேட்டா முழுசும் அழிஞ்சிடும்.

ஆனா, சர்வர் ரீஸ்டார்ட் ஆனா உள்ள இருக்கிற செஷன் டேட்டா எல்லாம் காலி. சரியான அனாலஜி. அப்புறம் இபிஎஸ் அப்படிங்கிறது ட்ரெடிஷனல் ரிலேஷனல் டிபி [Relational DB - டேபிளா, ரிலேஷன்ஷிப்போட டேட்டா ஸ்டோர் பண்றது], டிஸ்க் மாதிரி. கொஞ்சம் நெட்வொர்க் ஜம்ப் இருக்கும், ஆனா டேட்டா சேஃபா, பெர்மனன்ட்டா இருக்கும். அப்போ, இன்ஸ்டன்ஸ் ஸ்டோரை நாம முக்கியமான டேட்டாவை சேமிக்க யூஸ் பண்ணவே கூடாது.

கண்டிப்பா கூடாது.

மாறாக, கேஷ் பஃபர்ஸ் [Cache buffers - டேட்டாவை வேகமா கொடுக்க டெம்பரரியா ஸ்டோர் பண்ற இடம்], ஸ்கிராட்ச் டேட்டா [Scratch data - டெம்பரரியா ப்ராசஸ் பண்ண யூஸ் பண்ற டேட்டா], அல்லது ப்ராசஸிங்க்கு மட்டும் யூஸ் பண்ற டெம்பரரி ஃபைல்ஸ்க்கு தான் யூஸ் பண்ணனும் இல்லையா?

மிக சரியான ஒப்பீடு. அதனால தான் என்டர்பிரைஸ் டேட்டாபேசஸை எப்பவுமே இபிஎஸ்ல தான் ரன் பண்ணுவாங்க. இபிஎஸ்லயே ஒர்க்லோட்ஸ்க்கு தகுந்த மாதிரி வகைகள் இருக்கு. ஜெனரல் பர்பஸ்க்கு gp2, gp3-ன்னு எஸ்டிடி [SSD - ரொம்ப வேகமா வேலை செய்யுற சாலிட் ஸ்டேட் டிரைவ் ஸ்டோரேஜ்] யூஸ் பண்ணுவாங்க.

மிஷன் கிரிட்டிக்கல் ஆப்ஸ்க்கு?

நீங்க ஒரு மிஷன் கிரிட்டிக்கலான ஆரக்கிள் டேட்டாபேஸை ரன் பண்றீங்க, அதுக்கு ஒரு செகண்டுக்கு இத்தனை முறை டேட்டாவை ரீட், ரைட் பண்ண முடியும் அப்படிங்கிற வேகம் மிக முக்கியம் அப்படின்னா, io1 அல்லது io2 பிளாக் எக்ஸ்பிரஸ் புரொவிஷன்ட் ஐஓபிஎஸ் வால்யூம்ஸை யூஸ் பண்ணுவாங்க.

சூப்பர். அப்போ இபிஎஸ் தான் சேஃப்னு முடிவு பண்ணியாச்சு. ஆனா, SAA எக்ஸாம்ல இத வச்சு ஒரு ட்விஸ்ட் வைப்பாங்களே? இபிஎஸ் ஒரு குறிப்பிட்ட அவைலபிலிட்டி ஜோன் [Availability Zone - ஒரு ரீஜனுக்குள்ள இருக்கிற தனித்தனி டேட்டா சென்டர்கள்] அதாவது AZ-ல மட்டும் தான் லாக் ஆகிருக்கும்.

ஆமா, அது ட்ரூ தான்.

ஒருவேளை US-East-1A அப்படிங்கிற அவைலபிலிட்டி ஜோன்ல என்னோட இபிஎஸ் வால்யூம் இருக்கு. அந்த டேட்டா சென்டர்ல ஒரு பெரிய பவர் அவுட்டேஜ் [Power outage - கரண்ட் கட் ஆகுறது] வருது. இப்போ அந்த டேட்டாவை நான் US-East-1B அப்படிங்கிற வேறொரு AZ-க்கு எப்படி மாற்றுவது? நெட்வொர்க் டிரைவை அப்படியே கழட்டி அங்க மாட்ட முடியாதே?

கண்டிப்பா முடியாது. ஏன்னா, இபிஎஸ் வால்யூம் எந்த ஏசட்ல இருக்கோ, அந்த டேட்டா சென்டர் நெட்வொர்க்ல மட்டும் தான் பிசிக்கலா கனெக்ட் ஆகியிருக்கும். இதை வேற ஏசட்டுக்கு கொண்டு போகணும்னா, முதல்ல அந்த இபிஎஸ் வால்யூமை ஒரு ஸ்னாப்ஷாட் [Snapshot - ஒரு குறிப்பிட்ட நேரத்துல டேட்டா எப்படி இருந்துச்சோ அத அப்படியே ஒரு பேக்கப் காப்பி எடுக்குறது] எடுக்கணும்.

ஸ்னாப்ஷாட்னா பேக்கப் [Backup - டேட்டாவை பத்திரமா இன்னொரு காப்பி எடுத்து வைக்கிறது] மாதிரி தானே?

ஆமா, ஸ்னாப்ஷாட் அப்படிங்கிறது அந்த வால்யூமோட பாயிண்ட் இன் டைம் பேக்கப். இந்த பேக்கப் AWS ஓட S3ல போய் ஸ்டோர் ஆகும். S3 ஒரு ரீஜன் லெவல் சர்வீஸ். அதனால, அந்த ஸ்னாப்ஷாட்டை வச்சு US-East-1B அப்படிங்கிற புது ஏசட்ல நீங்க ஒரு புது இபிஎஸ் வால்யூமை ரீஸ்டோர் பண்ணிக்கலாம்.

ஓ, இந்த மாதிரி தான் மூவ் பண்ணனுமா?

ஆமா, இது SAA எக்ஸாம்ல கட்டாயம் வரக்கூடிய ஒரு முக்கியமான டிசாஸ்டர் ரெக்கவரி சினாரியோ.

பிரில்லியண்ட். இப்போ நம்மகிட்ட IAM செக்யூரிட்டி இருக்கு, ஈசி2 இன்ஸ்டன்ஸ்ல ஜேவிஎம் ஓடுது, இபிஎஸ்ல டேட்டா சேஃபா இருக்கு. அடுத்த இன்ப்ராஸ்ட்ரக்சர் ஸ்கேல் அண்ட் காஸ்ட் [Scale and cost - ஆப்ப பெருசாக்குறதும் அதுக்கான செலவும்] பத்தி பார்ப்போம். நாம இப்போ பேசின அவைலபிலிட்டி ஜோன்ஸ், ரீஜன்ஸ் இதெல்லாம் ஒரு என்டர்பிரைஸ் ப்ரொடக்ஷன் டிப்ளாய்மென்ட்ல எப்படி வேலை செய்யுது?

AWS உலகெங்கிலும் குளோபல் இன்ப்ராஸ்ட்ரக்சரை வச்சிருக்காங்க. ரீஜன் அப்படிங்கிறது ஒரு பெரிய புவியியல் அமைப்பு. உதாரணத்துக்கு மும்பை ரீஜன். ஒவ்வொரு ரீஜனுக்குள்ளயும் குறைந்தபட்சம் மூணு அவைலபிலிட்டி ஜோன்ஸ் இருக்கும்.

அந்த அவைலபிலிட்டி ஜோன்ஸ்லாம் ஒரே பில்டிங்ல இருக்குமா?

இல்ல இல்ல, ஏசட் அப்படிங்கிறது சும்மா ஒரு பில்டிங் கிடையாது. ஒவ்வொரு ஏசட்டும் ஒன்று அல்லது அதற்கு மேற்பட்ட தனித்தனி டேட்டா சென்டர்ஸ். ஒரு ஏசட்டுக்கும் இன்னொரு ஏசட்டுக்கும் பல கிலோமீட்டர் இடைவெளி இருக்கும்.

வாவ், அப்போ நெட்வொர்க், பவர் எல்லாம் தனித்தனி தானா?

ஆமா, இவற்றுக்கே தனித்தனி பவர் சப்ளை, தனித்தனி நெட்வொர்க் இருக்கும். ஒரு ஏசட்ல வெள்ளமோ, தீ விபத்தோ நடந்தா, அது இன்னொரு ஏசட்டை பாதிக்காத வகையில முற்றிலும் ஐசோலேட்டடா அமைக்கப்பட்டிருக்கும்.

அதனால தான் ஹை அவைலபிலிட்டி கிடைக்குது.

கரெக்ட். அதனால, நீங்க உங்க ஜாவா அப்ளிகேஷனை ஈசி2 இன்ஸ்டன்சஸ் மூலமா பல ஏசட்களில் பிரிச்சு டெப்லாய் பண்ணும்போது, ஒரு முழு டேட்டா சென்டரே டவுன் ஆனாலும், உங்க அப்ளிகேஷன் ஹை அவைலபிலிட்டில ஓடிட்டு இருக்கும்.

ஆனா, ஒரு கவலை என்னன்னா, எல்லா இடத்துலயும் சர்வரை ஸ்பின் பண்ணா, கிளவுட்ல காஸ்ட் எக்கச்சக்கமா ஏறிடுமே. நம்ம பிசிக்கல் டேட்டா சென்டர்ல ஒரு தடவை ஹார்டுவேர் வாங்கிட்டா முடிஞ்சது. ஆனா கிளவுட்ல பில் மீட்டர் ஓடிக்கிட்டே இருக்குமே? இதை AWS-ல எப்படி ஆப்டிமைஸ் பண்றது?

இதுக்காகவே AWS-ல மூணு முக்கியமான பர்ச்சேசிங் ஆப்ஷன்ஸ் இருக்கு. முதல்ல ஆன்-டிமாண்ட் [On-demand - எப்ப தேவையோ அப்போ மட்டும் யூஸ் பண்ணிட்டு காசு கொடுக்குறது]. நீங்க எப்போ சர்வர் யூஸ் பண்றீங்களோ, அந்த செகண்டுக்கு மட்டும் காசு கொடுத்தா போதும். புதுசா ஒரு அப்ளிகேஷன் டெஸ்ட் பண்றீங்க, லோடு எப்படி இருக்கும்னு தெரியலன்னா, ஆன்-டிமாண்ட் பெஸ்ட்.

ஓகே, அப்போ ஸ்டேபிளான லோடுக்கு என்ன பண்றது?

ரெண்டாவது, ரிசர்வ்ட் இன்ஸ்டன்சஸ் [Reserved instances - இவ்வளவு நாள் யூஸ் பண்ணுவேன்னு முன்னாடியே கமிட் பண்ணி டிஸ்கவுண்ட் வாங்குறது] அல்லது சேவிங்ஸ் பிளான்ஸ். உங்க என்டர்பிரைஸ் ஜாவா ஆப் பல வருஷமா ஒரே மாதிரியான ஸ்டேபிள் லோட்ல ஓடிட்டு இருக்குன்னு வைங்க. அடுத்த ஒரு வருஷம் அல்லது மூணு வருஷத்துக்கு நான் கண்டிப்பா இவ்வளவு கம்ப்யூட்ட யூஸ் பண்ணுவேன்னு AWS-க்கு ஒரு கமிட்மென்ட் கொடுத்தா, உங்களுக்கு 72% வரைக்கும் மிகப்பெரிய டிஸ்கவுண்ட் கிடைக்கும்.

ஓ, இது நியாயமா படுது. ரிசர்வ்ட் வாங்கிக்கலாம், திடீர்னு ஸ்பைக் வந்தா ஆன்-டிமாண்ட் யூஸ் பண்ணிக்கலாம். ஆனா ஸ்பாட் இன்ஸ்டன்சஸ் [Spot instances - AWS கிட்ட சும்மா இருக்கிற சர்வர்ஸ ரொம்ப கம்மி விலைக்கு வாங்குறது] அப்படின்னு ஒன்னு இருக்கே? அது ரொம்ப ரொம்ப சீப்னு கேள்விப்பட்டேன், 90% டிஸ்கவுண்ட் எல்லாம் தராங்களாமே? நிஜமா?

தராங்க, ஆனா இங்க ஒரு பெரிய கேட்ச் இருக்கு. ஸ்பாட் இன்ஸ்டன்சஸப் அப்படிங்கிறது AWS-கிட்ட சும்மா அன்யூஸ்டா தூங்கிட்டு இருக்குற ஹார்டுவேர். அதை அவங்க மிக குறைந்த விலைக்கு கொடுப்பாங்க.

ஓகே, அப்புறம் என்ன கேட்ச்?

கேட்ச் என்னன்னா, AWS-க்கு திடீர்னு அந்த கெபாசிட்டி தேவைப்பட்டா, வெறும் ரெண்டு நிமிஷம் வார்னிங் கொடுத்துட்டு உங்க ஈசி2 இன்ஸ்டன்ஸை அப்படியே உங்கள்ட்ட இருந்து புடுங்கிப்பாங்க.

ஒரு நிமிஷம், ரெண்டு நிமிஷத்துல சர்வரை பிடுங்கிடுவாங்களா? அப்போ ஸ்பாட் இன்ஸ்டன்சஸ வச்சு ஒரு ஜாவா டெவலப்பர் என்ன பண்றது? யூசர் ஒரு பெரிய பேமெண்ட் டிரான்சாக்ஷன் பண்ணிட்டு இருக்கும்போது சர்வர் காணாம போனா அப்ளிகேஷன் கிராஷ் ஆயிடுமே?

ஆமா, அதனாலதான் ஸ்பாட் இன்ஸ்டன்சஸை எந்த ஒரு ஸ்டேட்ஃபுல் அப்ளிகேஷனுக்கோ [Stateful application - யூசரோட டேட்டாவை அந்த சர்வர்லயே ஞாபகம் வச்சுக்கிற ஆப்] அல்லது மெயின் வெப் சர்வருக்கோ நீங்க கண்டிப்பா யூஸ் பண்ணக்கூடாது. உங்களோட ஆர்கிடெக்சரை நீங்க ஸ்டேட்லெஸ்ஸா [Stateless - எந்த யூசர் டேட்டாவையும் சர்வர்ல சேவ் பண்ணாம வெளியில வச்சுக்கிற ஆப்] டிசைன் பண்ணனும்.

ஸ்டேட்லெஸ்னா? செஷன் டேட்டாவை வெளியில வைக்கணுமா?

எக்ஸாக்ட்லி. யூசர் செஷன் டேட்டா எதையும் ஈசி2 இன்ஸ்டன்ஸ்ல வைக்காம, ரெடிஸ் மாதிரியான எக்ஸ்டர்னல் கேஷ்லயோ அல்லது டைனமோடிபிலயோ வச்சிடணும். ஒருவேளை AWS உங்க ஸ்பாட் இன்ஸ்டன்ஸை டெர்மினேட் பண்ணாலும், லோட் பேலன்சர் [Load balancer - வர்ற ட்ராபிக்கை எல்லா சர்வர்ஸ்க்கும் சமமா பிரிச்சு கொடுக்குறது] தானாகவே டிராபிக்கை வேறொரு ஈசி2க்கு அனுப்பிடும். யூசருக்கு எந்த பாதிப்பும் தெரியாது.

அப்போ ஸ்பாட் இன்ஸ்டன்சஸை எதுக்கு தான் யூஸ் பண்றது?

பேட்ச் ப்ராசஸிங் [Batch processing - நிறைய டேட்டாவை மொத்தமா சேர்த்து வச்சு ப்ராசஸ் பண்றது], பேக்ரவுண்ட் ஜாப்ஸ், இமேஜ் ரெண்டரிங் மாதிரியான ஃபெயிலியரை தாங்கக்கூடிய, அதாவது ஃபால்ட் டாலரண்ட் [Fault tolerant - இடையில ஏதாவது தப்பானாலும் நின்னுடாம தொடர்ந்து வேலை செய்யுறது] வேலைகளுக்கு மட்டும் தான் ஸ்பாட் இன்ஸ்டன்சஸ் யூஸ் பண்ணனும். SAA எக்ஸாம்ல மிக குறைந்த செலவில் ஃபால்ட் டாலரண்ட் ஒர்க்லோடை ரன் பண்ணனும்னு ஒரு கேள்வி வந்தா, கண்ணை மூடிக்கிட்டு ஸ்பாட் இன்ஸ்டன்சஸை டிக் பண்ணிடுங்க.

வாவ், உங்களோட இன்சைட்ஸ்ல இருந்து பல விஷயங்கள் ரொம்ப தெளிவா புரிஞ்சது. ஒரு 12 வருஷ ஜாவா அனுபவத்தை வச்சுக்கிட்டு கிளவுடை பார்க்கும்போது, IAM அப்படிங்கிறது நம்ம ஸ்ப்ரிங் செக்யூரிட்டி ஆர்பேக் கோடோட குளோபல் வெர்ஷன், அப்புறம் ஈசி2 அப்படிங்கிறது நாம டியூன் பண்ற பேர் மெட்டல் டாம்கேட் சர்வரோட அப்ஸ்ட்ராக்டட் இன்ப்ராஸ்ட்ரக்சர்.

ஆமா, அது மட்டுமில்லாம ஸ்டோரேஜையும் கனெக்ட் பண்ணலாம்.

கரெக்ட், இபிஎஸ் மற்றும் இன்ஸ்டன்ஸ் ஸ்டோர் அப்படிங்கிறது பெர்சிஸ்டன்ட் டிஸ்க் மற்றும் எஃபிமரல் மெமரிக்கான ட்ரேட் ஆஃப்ஸ் அப்படின்னு ஒரு தெளிவான மேப் நமக்கு கிடைச்சிருக்கு.

நிச்சயமா. கிளவுட் ஆர்கிடெக்சர் அப்படிங்கிறது ஏதோ புதுசு இல்ல, நாம ஏற்கனவே பண்ற டிஸ்ட்ரிபியூட்டட் சிஸ்டம்ஸ் டிசைனோட ஒரு மிக பிரம்மாண்டமான இம்ப்ளிமெண்டேஷன் தான். நீங்க எக்ஸிஸ்டிங் கான்செப்ட்ஸோட இப்படி ரிலேட் பண்ணி படிக்கும்போது, AWS இன்ப்ராஸ்ட்ரக்சர் உங்க மனசுல ஆழமா பதிஞ்சிடும்.

சூப்பரான டிஸ்கஷன். இறுதியா, இந்த விரிவான அலசலை கேட்டுட்டு இருக்கிற உங்களுக்கு ஒரு சிந்தனையை விட்டுட்டு போறேன். நாம இபிஎஸ் பற்றி பேசும்போது, ஒரு இபிஎஸ் வால்யூம் ஒரு அவைலபிலிட்டி ஜோன்ல மட்டும் தான் லாக் ஆகியிருக்கும்னு பார்த்தோம்.

ஆமா, அது ஒரு முக்கியமான பாயிண்ட்.

ஆனா, உங்களோட சோர்சஸ்ல ஈஎஃப்எஸ் (EFS), அதாவது எலாஸ்டிக் ஃபைல் சிஸ்டம் [EFS - நிறைய சர்வர்ஸ் ஒரே நேரத்துல ஷேர் பண்ணி யூஸ் பண்ற ஒரு நெட்வொர்க் ஃபைல் சிஸ்டம்] பத்தி ஒரு சின்ன குறிப்பு இருக்கு. இந்த ஈஎஃப்எஸ்ஸால 100க்கும் மேற்பட்ட ஈசி2 இன்ஸ்டன்சஸை, அதுவும் பல ஏசட்களில், ஒரே நேரத்துல மவுண்ட் பண்ண முடியும்.

அது ஒரு பவர்ஃபுல் ஃபீச்சர்.

ஒருவேளை நீங்க ஒரு டிஸ்ட்ரிபியூட்டட் ஜாவா அப்ளிகேஷனை டிசைன் பண்றீங்க, அதுல மூணு வெவ்வேறு ஏசட்களில் இருக்கிற டாம்கேட் இன்ஸ்டன்சஸ் எல்லாமே ஒரே ஷேர்ட் டைரக்டரில [Shared directory - எல்லாரும் ஒன்னா யூஸ் பண்ற ஒரு போல்டர்] ரீட் ரைட் பண்ணனும்னா, இபிஎஸ்ஸால அது முடியாது, ஆனா ஈஎஃப்எஸ்ஸால முடியும். அப்போ, நான் ஏன் எப்பவுமே ஈஎஃப்எஸ்ஸையே யூஸ் பண்ணக்கூடாது? எதுக்கு இபிஎஸ்க்கு போகணும் அப்படிங்கிற ஒரு கேள்வி வருது இல்லையா? அந்த நெட்வொர்க் லேட்டன்சி, ஃபைல் சிஸ்டம் லாக்ஸ், மற்றும் காஸ்ட் கன்ஸ்ட்ரெயின்ட்ஸ்ல என்னென்ன ட்ரேட் ஆஃப்ஸ் இருக்கு அப்படின்னு சொந்தமா கொஞ்சம் தேடி பாருங்க. உங்களோட ஆர்கிடெக்சர் முடிவுகளை அது அடுத்த கட்டத்துக்கு கொண்டு போகும். ஹேப்பி லேர்னிங் அண்ட் ஆல் தி பெஸ்ட் ஃபார் யுவர் SAA எக்ஸாம்.
