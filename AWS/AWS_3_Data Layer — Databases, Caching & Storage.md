ஸ்பீக்கர் 1: கண்டிப்பா சர்வர் திணறிடும். உங்க ஜேவிஎம் மெமரி [JVM memory - ஜாவா ப்ரோக்ராம் ஓடுறதுக்கு தேவைப்படுற ரேம்] ஃபுல்லாகி, அவுட் ஆஃப் மெமரி எரர் [Out of Memory error - ரேம் பத்தாம போயிருச்சுன்னு சிஸ்டம் காட்டுற எரர்] வந்து, சர்வர் அப்படியே க்ராஷ் ஆயிடும் [Crash - சர்வர் வேலை செய்யாம நின்னுடுறது].

S3-ய ஃபைல் சிஸ்டம்னு சொல்றது டெக்னிக்கலா தப்பு. அது ஒரு ஆப்ஜெக்ட் ஸ்டோரேஜ் [Object Storage - ஃபைல்ஸ போல்டரா வைக்காம, ஒவ்வொன்னையும் தனித்தனி ஆப்ஜெக்ட்டா ஐடியோட சேவ் பண்ற முறை]. நம்ம ஈசி2-ல பாத்த இபிஎஸ் (EBS) வந்து பிளாக் ஸ்டோரேஜ் [Block Storage - ஹார்ட் டிஸ்க்ல டேட்டாவை சின்ன சின்ன பிளாக்கா பிரிச்சு சேவ் பண்றது]. ஆனா S3 அப்படிங்கறது ஒரு பெரிய கீ-வேல்யூ ஸ்டோர் [Key-Value store - ஒரு பேரை வச்சு அதுக்குள்ள டேட்டாவை சேவ் பண்ற முறை] மாதிரி. ஃபைலோட பேரு வந்து கீ, உள்ள இருக்குற டேட்டா வந்து வேல்யூ. இது பக்கெட்ஸ் [Buckets - S3-ல ஃபைல்ஸ போட்டு வைக்கிற ஒரு மெய்நிகர் பை] அப்படிங்கற கோப்பைகளுக்குள்ள சேமிக்கப்படும்.

ஸ்பீக்கர் 2: அடிக்கடி யூஸ் பண்ற டேட்டாவ S3 ஸ்டாண்டர்ட் [S3 Standard - அடிக்கடி யூஸ் பண்ற ஃபைல்ஸ வைக்கிற நார்மல் S3 ஸ்டோரேஜ்] கிளாஸ்ல வைப்போம். ஆனா அடிக்கடி எடுக்க மாட்டோம், ஆனா கேட்ட உடனே வேணும்னா, S3 Standard-IA [S3 Standard-Infrequent Access - எப்பவாவது ஒரு தடவை தேவைப்படுற ஆனா கேட்ட உடனே கிடைக்க வேண்டிய டேட்டாவை சேவ் பண்ற இடம்] யூஸ் பண்ணலாம். பழைய லாக் ஃபைல்ஸ், பேக்கப்க்கு கிளேசியர் [Glacier - ரொம்ப வருஷத்துக்கு பேக்கப் வைக்கிற ரொம்ப சீப்பான ஸ்டோரேஜ்] கிளாஸஸ் இருக்கு.

ஸ்பீக்கர் 1: எந்த டேட்டாவ எப்போ யூஸ் பண்ணுவாங்கனு எனக்கு முன்னாடியே தெரியலைன்னா என்ன பண்றது?

ஸ்பீக்கர் 2: அதுக்குத்தான் S3 இன்டெலிஜென்ட் டையரிங் [S3 Intelligent-Tiering - ஃபைல எப்போ யூஸ் பண்றாங்கனு அதுவே செக் பண்ணி, காசு கம்மியாகுற மாதிரி வேற வேற ஸ்டோரேஜ்க்கு அதுவே மாத்துற ஃபீச்சர்] இருக்கு. அதுவே மானிட்டர் பண்ணி தானா மாத்திக்கும். இது தவிர லைஃப் சைக்கிள் ரூல்ஸ் [Lifecycle rules - இத்தனை நாளுக்கு அப்புறம் ஃபைல இந்த இடத்துக்கு மாத்துனு ஆட்டோமேட்டிக்கா செட் பண்ற ரூல்ஸ்] யூஸ் பண்ணி பழைய ஃபைல்ஸ ஆட்டோமேட்டிக்கா எக்ஸ்பிரேஷன் ஆக்சன் [Expiration action - குறிப்பிட்ட நாளுக்கு அப்புறம் ஃபைல ஆட்டோமேட்டிக்கா டெலீட் பண்ற செட்டிங்] மூலம் டெலீட் பண்ணலாம்.

ஸ்பீக்கர் 1: சரி, இப்போ S3 பாத்தாச்சு. ஆனா ஒரு அப்ளிகேஷனோட இதயமே ரிலேஷனல் டேட்டாபேஸ் தான். மைஎஸ்கியூஎல் (MySQL), போஸ்ட்கிரஸ் (Postgres) மாதிரி டேட்டாபேஸ்க்கு க்ளவுட்ல என்ன தீர்வு?

ஸ்பீக்கர் 2: அங்கதான் ஆர்டிஎஸ் (RDS) அதாவது ரிலேஷனல் டேட்டாபேஸ் சர்வீஸ் [Relational Database Service - ரிலேஷனல் டேட்டாபேஸ்களை ரன் பண்ணி மேனேஜ் பண்ற AWS சர்வீஸ்] வருது. இது ஒரு மேனேஜ்ட் சர்வீஸ் [Managed service - சர்வர் மெயின்டனன்ஸ் எல்லாமே கம்பெனியே பாத்துக்குற சர்வீஸ்]. பாயிண்ட்-இன்-டைம் ரீஸ்டோர் [Point-in-time restore - ஒரு குறிப்பிட்ட நேரத்துல டேட்டாபேஸ் எப்படி இருந்துச்சோ அப்படியே டைம் மெஷின் மாதிரி பின்னாடி கொண்டு போறது] ஆப்ஷன்லாம் இதுல இருக்கு.

ஸ்பீக்கர் 1: மெயின் டேட்டாபேஸ் க்ராஷ் ஆனா என்ன பண்றது? ஹை அவைலபிலிட்டிக்கு என்ன வழி?

ஸ்பீக்கர் 2: அதுக்கு மல்டி-ஏஜி டிப்ளாய்மென்ட் [Multi-AZ deployment - ஒரு டேட்டா சென்டர் டவுன் ஆனாலும் இன்னொரு சென்டர்ல இருந்து ஆப் வேலை செய்யுற மாதிரி செட் பண்றது] யூஸ் பண்ணனும். இதுல சின்க்ரனஸ் ரெப்ளிகேஷன் [Synchronous replication - மெயின் டிபிலயும் பேக்கப் டிபிலயும் ஒரே நேரத்துல டேட்டாவை ரைட் பண்ற முறை] நடக்கும். மெயின்ல ரைட் ஆகுற அதே செகண்ட்ல ஸ்டாண்ட்-பைலயும் [Standby copy - மெயின் சர்வர் டவுன் ஆனா யூஸ் பண்றதுக்கு தயாரா இருக்குற பேக்கப் சர்வர்] ரைட் ஆகும்.

ஸ்பீக்கர் 1: ரீட் டிராபிக் ரொம்ப அதிகமா இருந்தா? மாஸ்டர் டேட்டாபேஸ் திணறுமே?

ஸ்பீக்கர் 2: அதுக்கு ரீட் ரெப்ளிக்காஸ் [Read replicas - மெயின் டேட்டாபேஸ்ல இருந்து டேட்டாவை மட்டும் ரீட் பண்றதுக்காக உருவாக்கப்பட்ட டூப்ளிகேட் காப்பிகள்] கிரியேட் பண்ணனும். இதுல ஏ-சின்க்ரனஸ் ரெப்ளிகேஷன் [Asynchronous replication - மெயின் டிபில ரைட் ஆன உடனே சக்ஸஸ் சொல்லிட்டு, அப்புறமா பேக்கப் டிபிக்கு டேட்டாவை அனுப்புற முறை] நடக்கும்.

ஸ்பீக்கர் 1: ஓகே, ஆனா பாரம்பரிய டேட்டாபேஸ்ல ஒரு லிமிட்டுக்கு மேல ஸ்கேல் பண்ண முடியாது. அதுக்கு AWS-ல அமேசான் அரோரா [Amazon Aurora - AWS சொந்தமா உருவாக்குன ரொம்ப ஸ்பீடான, க்ளவுட்க்கு ஏத்த மாதிரி டிசைன் பண்ணப்பட்ட ரிலேஷனல் டேட்டாபேஸ்] இருக்கே.

ஸ்பீக்கர் 2: சரியா சொன்னீங்க. அரோரால கம்ப்யூட் லேயரையும் ஸ்டோரேஜ் லேயரையும் முழுசா பிரிச்சிட்டாங்க. நெட்வொர்க் I/O [Network I/O - நெட்வொர்க் வழியா டேட்டாவை ரீட் அல்லது ரைட் பண்ற வேலை] குறைவு, அதனால சாதாரண டேட்டாபேஸை விட 5 மடங்கு வேகம்.

ஸ்பீக்கர் 1: சரி, டேட்டாபேஸ் லோடை குறைக்க லோக்கல் ஜேவிஎம் கேச் யூஸ் பண்ணலாமா?

ஸ்பீக்கர் 2: க்ளவுட்ல ஆட்டோ ஸ்கேலிங் நடக்கும்போது பல சர்வர் ஓடும். லோக்கல் கேச் வச்சா டேட்டா இன்கன்சிஸ்டன்சி [Data inconsistency - ஒரு இடத்துல ஒரு டேட்டாவும், இன்னொரு இடத்துல பழைய டேட்டாவும் இருக்குற முரண்பாடான நிலை] வந்துடும். அதனால சென்ட்ரலைஸ்டு கேச் வேணும்.

ஸ்பீக்கர் 1: அதுக்குதான் அமேசான் எலாஸ்டிக் கேச் [Amazon ElastiCache - மெமரிலயே டேட்டாவை சேவ் பண்ற வேகமான க்ளவுட் கேஷிங் சர்வீஸ்] யூஸ் பண்ணனுமா?

ஸ்பீக்கர் 2: ஆமா, ரெடிஸ் [Redis - ரொம்ப ஃபாஸ்ட்டா வேலை செய்யுற இன்-மெமரி டேட்டாபேஸ்] அல்லது மெம்கேஷ்டு யூஸ் பண்ணலாம். இது லேசி லோடிங் [Lazy loading - முதல்ல கேச்ல தேடிட்டு, அங்க இல்லனா மட்டும் டேட்டாபேஸ்க்கு போய் டேட்டாவை எடுக்குற முறை] பேட்டர்ன்ல சப்-மில்லிசெகண்ட் லேட்டன்சில [Sub-millisecond latency - ஒரு செகண்ட்டோட ஆயிரத்துல ஒரு பங்குக்கும் குறைவான நேரத்துல ரெஸ்பான்ஸ் கிடைக்கிறது] டேட்டாவை கொடுக்கும்.

ஸ்பீக்கர் 1: கேமிங் லீடர்போர்ட்ஸ் [Gaming leaderboards - கேம்ல யார் ஃபர்ஸ்ட் இருக்காங்கன்னு காட்ற ஸ்கோர் போர்டு] மாதிரி ரியல் டைம் ரேங்கிங்க்கு ரெடிஸ்ல என்ன ஃபீச்சர் இருக்கு?

ஸ்பீக்கர் 2: ரெடிஸ்ல சார்ட்டட் செட்ஸ் [Sorted Sets - டேட்டாவை சேர்க்கும்போதே ஆட்டோமேட்டிக்கா வரிசைப்படுத்தி வச்சுக்கிற ரெடிஸ் டேட்டா ஸ்ட்ரக்சர்] இருக்கு. இதை யூஸ் பண்ணி மைக்ரோசெகண்ட்ஸ்ல லீடர்போர்ட் அப்டேட் பண்ணிடலாம். ஆனா கேச் இன்வேலிடேஷன் [Cache invalidation - டேட்டாபேஸ்ல டேட்டா மாறும்போது, கேச்ல இருக்குற பழைய டேட்டாவை டெலீட் பண்ற லாஜிக்] லாஜிக்க நாமதான் கரெக்டா எழுதணும்.

ஸ்பீக்கர் 1: சூப்பர். ஃபைனலா அந்த AI ஃபீச்சர் பத்தி சொல்லுங்க.

ஸ்பீக்கர் 2: ஆமா, அரோரா மெஷின் லேர்னிங் [Aurora Machine Learning - டேட்டாபேஸ்க்குள்ளயே AI மாடல்ஸ வச்சு டேட்டாவை பிரடிக்ட் பண்ற ஃபீச்சர்] வந்துருக்கு. டேட்டாபேஸே ஒரு அறிவாளியா மாறி யூசர் பிஹேவியர பிரடிக்ட் பண்ற காலம் வந்துருச்சு. 

ஸ்பீக்கர் 1: ரொம்ப அருமையான செஷன். ஃபைல்ஸ்க்கு S3, ஸ்ட்ரக்சர்டு டேட்டாவுக்கு [Structured data - டேபிள்ல ரோ, காலமா நீட்டா அடுக்கி வச்சிருக்கற டேட்டா] RDS அல்லது அரோரா, பெர்பார்மன்ஸ்க்கு எலாஸ்டிக் கேச். நல்லா புரிஞ்சது. அடுத்த விரிவான அலசல்ல சந்திப்போம். நன்றி!
